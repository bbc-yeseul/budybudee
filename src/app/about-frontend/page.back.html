프론트엔드가 시안을 볼 때

디자이너와 개발자는 같은 시안을 다른 눈으로 봐요.

✅ 개발자가 좋아하는 시안
✓
컴포넌트 단위로 일관된 스타일
버튼, 카드가 전 페이지에서 동일하면 한 번 만들어두고 재사용 가능해요.
재사용 가능 · 속도 ↑
✓
8px 그리드 기반 여백
여백이 8, 16, 24px처럼 배수로 떨어지면 CSS로 옮기기 쉬워요.
구현 용이 · 일관성 ↑
✓
빈 상태·에러 화면 포함
데이터가 없을 때, 에러날 때 화면이 시안에 있으면 누락 없이 구현돼요.
예외처리 명확
✓
모바일 시안도 함께
PC만 있으면 모바일을 개발자가 스스로 판단해야 해요.
반응형 정확도 ↑
❌ 개발자가 힘든 시안
✗
페이지마다 다른 버튼 스타일
재사용이 안 되고 코드가 복잡해져요.
컴포넌트 불가 · 작업량 ↑
✗
픽셀 단위 절대 위치
"좌상단에서 137px" 같은 고정 위치는 화면 크기 바뀌면 무너져요.
반응형 불가
✗
텍스트 항상 1줄 고정
실제 데이터는 길이가 달라요. 긴 텍스트 대응이 없으면 레이아웃이 터져요.
예외 케이스 누락
✗
인터랙션 설명 없는 시안
hover, 비활성화 상태가 없으면 개발자가 전부 추측해야 해요.
커뮤니케이션 비용 ↑
collaboration tips

타 직군과 일할 때 꿀팁

이렇게 해주시면 정말 일하기 좋아요.

기획자
  • ✓버튼 클릭 후 동작을 명세서에 써줘요
  • ✓예외 케이스(빈 값, 에러)도 정의해줘요
  • ✓기능 변경은 문서나 티켓으로 남겨줘요
  • ✓"될 것 같아요?" 보다 "언제까지 가능해요?"
디자이너
  • ✓색상·폰트를 Figma 변수로 정의해줘요
  • ✓hover·focus·disabled 상태도 넣어줘요
  • ✓컴포넌트 단위로 프레임을 나눠줘요
  • ✓"느낌이 달라요"보다 "이 값이 달라요"
퍼블리셔
  • ✓클래스 이름을 기능 기반으로 지어줘요
  • ✓인라인 스타일보다 클래스로 처리해줘요
  • ✓JS 붙을 곳에 id나 data 속성 달아줘요
  • ✓반복 구조를 미리 얘기해줘요
백엔드
  • ✓API 명세서(Swagger)를 미리 공유해줘요
  • ✓에러 응답에 이유를 담아줘요 (message)
  • ✓API 변경사항은 사전에 알려줘요
  • ✓개발 서버와 목업 데이터를 같이 올려줘요
💡프론트도 마찬가지예요. 이상한 거 있으면 슬랙 DM 주세요!
frequently asked

"왜 그게 안 돼요?" — 자주 받는 질문

다른 직군에서 프론트엔드한테 자주 물어보는 것들이에요.

Q.디자인대로 만들었는데 왜 다르게 보여요?▼
브라우저마다 기본 스타일이 달라요. Figma는 픽셀 단위로 정확하지만 웹은 화면 크기가 제각각이라 폰트, 줄바꿈 위치가 달라져요. 이걸 맞추는 게 프론트 작업의 상당 부분이에요.
Q.이거 구현하는 데 왜 이렇게 오래 걸려요?▼
화면이 단순해 보여도 안에서 처리하는 게 많아요. "검색 필터" 하나만 해도 — 입력값, API 요청, 로딩, 에러, 빈 결과, URL 동기화, 모바일 레이아웃까지. 눈에 보이는 건 10%예요.
Q.프론트엔드가 데이터베이스도 건드려요?▼
일반적으로는 아니에요. DB는 백엔드 영역이고 프론트는 API를 통해서만 데이터를 받아요. Next.js 같은 풀스택 프레임워크를 쓰면 경계가 흐려지기도 해요.
Q."새로고침하면 된다"는 말을 왜 자주 해요?▼
브라우저가 이전에 받아둔 캐시를 그대로 써서 새 코드가 반영 안 되는 경우예요. 새로고침하면 최신 파일을 다시 받아와서 해결돼요.
Q.앱이랑 웹이랑 뭐가 달라요?▼
웹은 브라우저에서, 앱은 설치해서 사용해요. 프론트엔드는 주로 웹을 만들어요. React Native, Flutter로 앱을 만들기도 하고, 하나의 코드로 둘 다 만드는 시도도 많아요.
glossary

용어 사전

회의할 때 갑자기 튀어나오는 단어들. 클릭하면 펼쳐져요.

🔌 API / 통신
API백엔드와 대화하는 창구▼
프론트가 "이 유저 정보 줘"라고 요청하면 백엔드가 데이터를 돌려주는 약속된 통로예요.
GET /api/users/123 → { name: "김철수", age: 28 }
REST API가장 흔한 API 방식▼
URL로 대상을 표현하고, GET·POST·DELETE 동사로 행동을 표현해요.
GET /posts → 목록 POST /posts → 생성 DELETE /posts/5 → 삭제
JSON데이터 교환 포맷▼
프론트와 백엔드가 데이터를 주고받을 때 쓰는 포맷이에요.
{ "name": "이영희", "role": "designer" }
HTTP 상태코드요청 결과를 숫자로 표현▼
서버가 응답할 때 결과를 숫자로 알려줘요.
200 OK → 성공 404 Not Found → 없는 주소 401 Unauthorized → 로그인 필요 500 Server Error → 서버 문제
CORS도메인 간 요청 제한 정책▼
다른 도메인에서 API를 마음대로 못 부르게 막는 브라우저 정책이에요. 백엔드에서 허용 설정을 해줘야 해결돼요.
async/await기다리는 코드 작성법▼
API 호출처럼 시간이 걸리는 작업 앞에 붙여요. 결과가 올 때까지 기다렸다가 다음 줄을 실행해요.
const data = await fetch('/api/users') const json = await data.json()
💻 코드 / 개념
컴포넌트재사용 가능한 UI 조각▼
버튼, 카드 같은 UI 요소를 레고 조각처럼 만들어 이곳저곳에 갖다 쓸 수 있어요.
상태 (State)화면이 기억하는 데이터▼
"로그인 됐나요?" 같은 현재 상황을 기억하는 것. 상태가 바뀌면 화면이 자동으로 다시 그려져요.
const [isLoggedIn, setIsLoggedIn] = useState(false)
렌더링화면을 그리는 것▼
HTML·CSS·JS를 브라우저가 해석해서 눈에 보이는 화면으로 그려내는 과정이에요.
CSR / SSR화면을 어디서 그리냐▼
CSR은 브라우저에서, SSR은 서버에서 HTML을 완성해서 내려줘요. SSR이 SEO와 초기 속도에 유리해요.
CSR: 빈 HTML → JS 실행 → 화면 완성 SSR: 서버에서 완성 HTML → 즉시 표시
빌드 (Build)배포 전 최적화 과정▼
개발용 코드를 브라우저가 빠르게 읽도록 압축·변환하는 작업. 배포 전에 꼭 해요.
npm run build → /dist 에 최적화 파일 생성
캐싱 (Cache)데이터 임시 저장▼
한 번 받아온 파일을 저장해뒀다가 재사용하는 것. "캐시 지워봐"라는 말이 나오는 이유예요.
라우팅URL에 따라 화면 이동▼
주소창의 URL이 바뀔 때 어떤 화면을 보여줄지 결정하는 것. 페이지를 새로 불러오지 않고도 화면이 바뀌는 게 SPA 라우팅이에요.
/ → 홈 화면 /login → 로그인 화면 /post/42 → 42번 글 상세
반응형화면 크기에 따라 자동으로 달라지는 레이아웃▼
PC에서 볼 때와 모바일에서 볼 때 화면 구성이 달라지는 것. CSS 미디어쿼리로 "화면이 768px 이하면 이렇게 보여줘"라고 지정해요.
@media (max-width: 768px) { /* 모바일에서만 적용 */ .sidebar { display: none; } }
🌐 크로스 브라우징 / 환경
크로스 브라우징어느 브라우저에서나 같게 보이게▼
크롬, 사파리, 파이어폭스, 엣지 등 브라우저마다 HTML·CSS를 해석하는 방식이 조금씩 달라요. "크롬에선 되는데 사파리에서 깨져요"가 바로 이 문제예요. 각 브라우저에서 테스트하고 맞춰주는 작업을 크로스 브라우징이라고 해요.
브라우저웹을 보는 프로그램▼
크롬, 사파리, 파이어폭스, 엣지처럼 웹사이트를 열어볼 수 있는 프로그램이에요. 프론트엔드 코드는 이 안에서 실행돼요. 브라우저마다 조금씩 다르게 동작하는 게 프론트 개발의 어려움 중 하나예요.
개발자 도구 (F12)브라우저 안의 개발자 전용 패널▼
F12를 누르면 열리는 패널이에요. 에러 메시지 확인, 네트워크 요청 확인, 특정 요소의 CSS 수정 등을 할 수 있어요. 프론트 개발자의 가장 친한 친구예요.
F12 → Console : 에러·로그 확인 F12 → Network : API 요청 확인 F12 → Elements: HTML/CSS 실시간 수정
콘솔 (Console)코드 실행 결과를 찍어보는 곳▼
F12 → Console 탭에서 볼 수 있어요. 코드가 원하는 값을 갖고 있는지 확인하거나, 에러 메시지를 읽을 때 써요. 빨간 글씨가 뜨면 에러가 난 거예요.
console.log('값 확인:', user) // → F12 콘솔에서 실시간으로 볼 수 있어요
localhost내 컴퓨터에서만 열리는 개발용 주소▼
개발할 때 실제 서버에 올리기 전에 내 컴퓨터에서 먼저 테스트해요. 이때 주소창에 뜨는 게 localhost예요. 다른 사람 컴퓨터에서는 열 수 없어요.
http://localhost:3000 ← 내 컴퓨터에서만 접속 가능 https://myapp.com ← 배포 후 누구나 접속 가능
배포 (Deploy)만든 걸 실제 인터넷에 올리는 것▼
개발이 끝난 코드를 실제 사용자가 접속할 수 있는 서버에 올리는 과정이에요. "배포 나갔어요"는 실제 서비스에 반영됐다는 뜻이에요.
🛠 개발 도구 / 협업
Git코드 변경 이력을 관리하는 도구▼
코드를 언제, 누가, 무엇을 바꿨는지 기록해두는 도구예요. 실수로 코드를 망가뜨려도 이전 상태로 되돌릴 수 있어요. 팀원들과 코드를 합치는 것도 Git으로 해요.
git commit → 변경사항 저장 (세이브 포인트) git push → 원격 저장소에 올리기 git pull → 팀원 코드 받아오기
PR (Pull Request)코드 리뷰 요청▼
"내가 만든 기능 코드를 메인 코드에 합쳐도 될까요? 먼저 검토해줘요"라는 요청이에요. 팀원들이 코드를 보고 피드백을 남기면, 문제없으면 합쳐요(merge).
npm / yarn패키지 설치 도구▼
다른 개발자가 만들어놓은 코드 묶음(패키지, 라이브러리)을 프로젝트에 설치하는 도구예요. 앱스토어처럼 필요한 걸 골라서 설치해요.
npm install react → React 설치 npm run dev → 개발 서버 실행 npm run build → 배포용 파일 만들기
라이브러리 / 프레임워크남이 만들어둔 코드 도구 모음▼
처음부터 다 만들 필요 없이 남이 잘 만들어둔 걸 가져다 쓰는 것. 라이브러리는 필요한 것만 골라 쓰는 도구 모음이고, 프레임워크는 개발 방식 자체를 정해주는 큰 틀이에요. React는 라이브러리, Next.js는 프레임워크예요.
환경변수코드 밖에서 관리하는 민감한 설정값▼
API 키, 서버 주소처럼 코드에 직접 쓰면 안 되는 값들을 별도 파일(.env)에 저장해두는 방식이에요. GitHub에 올릴 때도 이 파일은 빼고 올려요.
# .env 파일 (비공개) VITE_API_URL=https://api.myapp.com VITE_SECRET_KEY=비밀키1234
번들러 (Vite, Webpack)파일들을 합치고 최적화하는 도구▼
개발할 때 파일을 수백 개로 나눠 작성하지만, 브라우저에 보낼 땐 합쳐서 압축해야 빨라요. 이 작업을 자동으로 해주는 게 번들러예요. 요즘은 빠른 Vite를 많이 써요.
💬 회의·협업에서 자주 나오는 말
QA배포 전 품질 검사▼
Quality Assurance의 줄임말. 개발이 끝난 기능을 배포하기 전에 버그는 없는지, 기획대로 잘 작동하는지 테스트하는 과정이에요. "QA 중이에요"는 아직 테스트 단계라는 뜻이에요.
스프린트정해진 기간 단위로 일하는 방식▼
보통 1~2주 단위로 목표를 정하고 그 안에 개발을 완료하는 애자일 방식의 개발 주기예요. "이번 스프린트에 넣자"는 이번 1~2주 안에 개발하자는 뜻이에요.
핫픽스긴급 버그 수정 배포▼
서비스에 심각한 버그가 생겼을 때 정기 배포를 기다리지 않고 급하게 수정해서 올리는 것. "핫픽스 나갑니다"는 긴급 상황이에요.
레거시 (Legacy)오래되고 낡은 코드▼
오래 전에 짠 코드라 지금 기준으로는 비효율적이지만 건드리기 무서운 코드예요. "레거시가 많아서요..."는 기술 부채가 쌓여있다는 뜻이에요.
기술 부채나중에 갚아야 할 코드의 빚▼
빠르게 개발하느라 제대로 짜지 않은 코드들이 쌓이는 것. 당장은 빠르지만 나중에 고치거나 기능을 추가할 때 더 오래 걸리게 돼요.
리팩토링기능은 그대로, 코드만 다시 정리▼
화면에 보이는 결과는 똑같지만 코드 내부를 더 깔끔하고 효율적으로 다시 짜는 것. "리팩토링 좀 해야 해요"는 기능 말고 코드 정리가 필요하다는 뜻이에요.
스택 오버플로 / 구글링개발자의 필수 검색 활동▼
Stack Overflow는 개발자들이 질문하고 답하는 글로벌 커뮤니티예요. 에러가 나거나 모르는 게 있으면 구글에 검색해서 Stack Overflow 글을 보는 게 기본이에요. 모든 개발자가 해요.
